       Both derive from the existing Default, so they initialize to 0.

    2. Add use statement for the git_state service:

    3. In run_full_inner, after the existing discovery transaction commits (after tx.commit()?;
       at line 150), add the git refresh second pass:

       a. Query all non-excluded repo paths:
          let all_paths: Vec<PathBuf> = {
              let mut stmt = conn.prepare("SELECT path FROM repos WHERE excluded = 0")?;
              stmt.query_map([], |row| row.get::<_, String>(0))?
                  .filter_map(|r| r.ok())
                  .map(PathBuf::from)
                  .collect()
          };

       b. Run parallel refresh (outside any DB borrow — rayon must complete before opening tx):
          let git_results = git_state::refresh_all(all_paths);

       c. Count results for summary:
          for r in &git_results {
              if r.state.error.is_some() {
                  summary.git_errors += 1;
              } else {
                  summary.git_refreshed += 1;
              }
          }

       d. Write results in a single transaction (Pitfall 6 — must be separate from discovery tx):
          let git_tx = conn.unchecked_transaction()?;
          let refresh_time = now_secs();
          for r in &git_results {
              git_tx.execute(
                  "UPDATE repos SET branch=?1, is_dirty=?2, ahead=?3, behind=?4,
                                    git_refreshed_at=?5, git_state_error=?6
                   WHERE path=?7",
                  rusqlite::params![
                      r.state.branch,
                      r.state.is_dirty.map(|b| b as i64),
                      r.state.ahead,
                      r.state.behind,
                      refresh_time,
                      r.state.error,
                      r.path,
                  ],
              )?;
          }
          git_tx.commit()?;

       Note: refresh_time is set once before the loop (same timestamp for all repos in a batch).
       is_dirty must be cast as Option<i64>: None stays NULL; Some(true) → 1; Some(false) → 0.

    4. The git refresh stats (git_refreshed, git_errors) do NOT go into index_runs table (per RESEARCH.md
       deferred decision — structured git stats in index_runs is deferred; Phase 3 outputs them
       to CLI only). Do not modify finish_index_run or the index_runs INSERT.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite repos table → git_tx UPDATE | Batch UPDATE writes git state from git2 back to DB — no external input, internal data only |
| DB path → workpot index | Paths from DB passed to refresh_all — canonicalized in refresh_git_state (T-03-01 from Plan 01) |
| CLI output | format_git_state formats DB data — no injection risk for local CLI output |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Tampering | index.rs git_tx batch UPDATE — paths from DB used as WHERE clause | accept | paths are parameterized (rusqlite::params![]) not interpolated; SQL injection not possible |
| T-03-08 | Denial of Service | refresh_all on large repo sets (>1000 repos) | accept | rayon default pool uses available cores; 1000 repos estimated at ~200ms on 4-core; D-23 cap is 20000 hard max but default is 1000 — acceptable |
| T-03-09 | Tampering | git_state_error stored and displayed in CLI output | accept | Error messages sourced from git2 C library (not user input); local use only; no injection vector |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | All three crates (git2, rayon, humantime) were verified by checkpoint in Plan 01 Task 1; no new installs in Plan 03 |
</threat_model>
