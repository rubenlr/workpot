## User Constraints (from CONTEXT.md)

### Locked Decisions

**Path template resolution**
- D-01: `{project}` resolves from global setting `migration.project_name_source`: `folder_name` (default) or `alias`. Fall back to folder name when alias is selected but absent.
- D-02: Default bare layout templates (global, in `config.toml`): bare repo `{project}/bare.git`, worktree `{project}/wtrees/{worktree}` (ROADMAP example A).
- D-03: Path templates are global only in `config.toml` — no per-repo template overrides in v1.
- D-04: `{worktree}` sanitization: map `/` → `.` in branch names; on path collision after sanitization, append a short hash suffix to disambiguate.

**Sync preflight gate**
- D-05: "Clean" means no dirty state in any linked worktree — run git status semantics across all worktrees (single worktree for normal; all worktrees under bare for bare repos).
- D-06: "Fully synced" is strict: every local branch must have an upstream configured AND `ahead == 0` on each branch (not only the current branch).
- D-07: Block conversion if any stash exists. Submodules ignored for v1 preflight.
- D-08: Bare → normal: target checkout is always the default branch (`main` / `master` detection), not the worktree path the user invoked convert on.

**Conversion clone strategy**
- D-09: Fresh clone source is the renamed local `.temp` directory (`git clone` from local path) — no network required.
- D-10: Normal → bare: first (only initial) worktree uses the currently checked-out branch at convert time.
- D-11: Health check is minimal: `git rev-parse HEAD` succeeds; for bare layout, `git worktree list` shows expected paths.
- D-12: On clone or health-check failure: leave `.temp` rename in place, abort, do not update DB — user recovers manually.

**Command surface & SETTINGS.md**
- D-13: CLI: `workpot repo convert <path> --to bare|normal` with clear preflight error messages per failed gate.
- D-14: `SETTINGS.md` at workpot repo root — user-facing documentation of all settings (including migration templates, temp suffix, delete-original, project name source) with ROADMAP examples.
- D-15: CLI only for v1 — no tray convert button or detail-pane action in this phase.
- D-16: Support `--dry-run`: print resolved target paths and preflight gate results; no filesystem or DB changes.
- D-17: `migration.temp_suffix` default `.temp` (always applied on rename; not optional).
- D-18: `migration.delete_original` default `false` — when true, delete renamed `.temp` after successful conversion; DB always drops old path regardless of filesystem delete.
- D-19: Bare → normal target path is `{project}` only (no worktree template segment).

### Claude's Discretion
- Exact `config.toml` key nesting (`[migration]` table shape), hash length for collision suffix, and default-branch detection order when both `main` and `master` absent.
- Whether `--dry-run` prints human table vs JSON (default human).
- Post-success catalog update: re-use `register_manual` / path-key update vs dedicated migration service (must preserve alias, tags, notes, pin).

## Summary

Phase 07.1 adds a `workpot repo convert <path> --to bare|normal` command backed by a new `services/repo_convert.rs` service. All necessary building blocks exist in the codebase: `git2` 0.21.0 is already vendored, `is_bare_repo`/`is_git_worktree`/`list_worktree_paths` are in `catalog.rs` and `infra/git.rs`, branch-level ahead/behind is in `infra/git.rs`, and the `Config` struct follows a well-established serde/default pattern for extension.

The three novel implementation challenges are: (1) the all-branches sync gate requires iterating `repo.branches(Some(BranchType::Local))` and checking upstream + ahead for every branch, plus `repo.stash_foreach` for stash detection; (2) the filesystem conversion sequence (rename → local clone → worktree add → health check) must be atomic in the failure direction — on any error, `.temp` stays in place and DB is untouched; (3) the catalog path-key swap must preserve all org fields (alias, tags, notes, pin) by using a DB-level path UPDATE rather than delete+reinsert.

TDD mode is active. The preflight gate functions (`run_preflight`, `resolve_path_template`, `sanitize_worktree_name`) have deterministic inputs and outputs and must be written test-first. The integration slice (convert a real temp git repo) drives the filesystem+catalog path through the full flow.

**Primary recommendation:** Implement as `crates/workpot-core/src/services/repo_convert.rs` orchestrating gate → rename → clone → health → catalog swap; add `Convert` variant to `RepoCommands` in CLI; extend `Config` with a `[migration]` section; produce `SETTINGS.md` at repo root.

### Pattern 2: All-branches sync gate using git2 0.21

[CITED: docs.rs/git2/0.21.0/git2/struct.Repository.html — branches() and Branch::upstream()]

```rust
// Source: git2 0.21.0 Repository::branches + Branch::upstream
pub fn check_all_branches_synced(repo: &Repository) -> Result<PreflightGateResult, WorkpotError> {
    let branches = repo.branches(Some(git2::BranchType::Local))
        .map_err(|_| WorkpotError::GitUnavailable(...))?;

    for item in branches {
        let (branch, _) = item.map_err(|_| WorkpotError::GitUnavailable(...))?;
        let name = branch.name()?.unwrap_or("?").to_string();

        // D-06: every local branch must have upstream AND ahead == 0
        let upstream = match branch.upstream() {
            Ok(u) => u,
            Err(_) => return Ok(PreflightGateResult::NoUpstream(name)),
        };
        let local_oid = branch.get().target()
            .ok_or_else(|| WorkpotError::GitUnavailable(...))?;
        let upstream_oid = upstream.get().target()
            .ok_or_else(|| WorkpotError::GitUnavailable(...))?;
        let (ahead, _behind) = repo.graph_ahead_behind(local_oid, upstream_oid)
            .map_err(|_| WorkpotError::GitUnavailable(...))?;
        if ahead > 0 {
            return Ok(PreflightGateResult::UnpushedCommits { branch: name, count: ahead });
        }
    }
    Ok(PreflightGateResult::Ok)
}
```

### Pattern 3: Stash detection using git2 0.21

[CITED: docs.rs/git2/0.21.0 — Repository::stash_foreach; callback signature `FnMut(usize, &str, &Oid) -> bool`]

```rust
// Source: git2 0.21 stash_foreach — note: takes &mut self
pub fn has_stash(repo: &mut Repository) -> Result<bool, WorkpotError> {
    let mut found = false;
    repo.stash_foreach(|_index, _msg, _oid| {
        found = true;
        false  // stop on first stash
    }).map_err(|_| WorkpotError::GitUnavailable(...))?;
    Ok(found)
}
```

**Important:** `stash_foreach` takes `&mut self` on `Repository`. The caller must open the repo with a mutable binding. This affects how preflight gathers both dirty check and stash check — open once, query both.

### Pattern 4: Catalog path-key swap preserving org fields

The `repos` table uses `path` as its PRIMARY KEY. Updating it in-place requires either:
- Option A: `UPDATE repos SET path=?new, name=?new_name, git_common_dir=?new_gcd WHERE path=?old` (all other columns preserved because we only update the named ones)
- Option B: DELETE + re-INSERT (loses all untouched columns — do NOT use this)

[CITED: SQLite documentation — UPDATE only named columns; repo_tags has FK ON DELETE CASCADE so tags move automatically if path is the FK parent]

```rust
// Catalog path-key swap — preserves alias, notes, pinned, pin_order, tags via FK cascade
conn.execute(
    "UPDATE repos SET path = ?1, name = ?2, git_common_dir = ?3,
     branch = NULL, is_dirty = NULL, ahead = NULL, behind = NULL,
     git_refreshed_at = NULL, git_state_error = NULL
     WHERE path = ?4",
    params![new_path_key, new_name, new_git_common_dir, old_path_key],
)?;
// repo_tags FK cascade: tag rows automatically get path = new_path_key? NO — FK cascade
// only fires on DELETE, not UPDATE of the parent PK.
// Must manually UPDATE repo_tags:
conn.execute(
    "UPDATE repo_tags SET repo_path = ?1 WHERE repo_path = ?2",
    params![new_path_key, old_path_key],
)?;
```

**Critical edge case:** `repo_tags` uses `repo_path` as a FK to `repos.path`. SQLite's ON DELETE CASCADE does NOT fire on UPDATE of the parent PK. Must explicitly update `repo_tags.repo_path` in a transaction alongside the repos UPDATE. [VERIFIED: SQLite docs — ON UPDATE NO ACTION is the default; CASCADE only applies if explicitly declared with ON UPDATE CASCADE which this schema does not use.]

### Pattern 5: Local git clone for bare layout

The conversion clone must be done via `std::process::Command` calling `git` (not git2) because git2's clone API does not expose `git clone --bare` + `git worktree add` as a single combined operation. This follows the established pattern in `crates/workpot-core/tests/common/mod.rs` where `seed_bare_repo` uses `git_cmd()`. [CITED: common/mod.rs — git_cmd() strips GIT_* env vars]

```rust
// Normal → bare: 3-step git sequence
// 1. git clone --bare <temp_dir> <bare_git_dir>
git_cmd().args(["clone", "--bare", "-q",
    temp_path.to_str()?, bare_git_path.to_str()?]).status()?;
// 2. Set push remote to point at itself (convention for bare repos used as worktree origin)
git_cmd().args(["remote", "set-url", "--push", "origin", "DISABLE"]).status()?;  // optional
// 3. git worktree add <worktree_path> <branch>
git_cmd().args(["worktree", "add", worktree_path.to_str()?, branch]).status()?;
```

For bare → normal:
```rust
// git clone <temp_bare_path> <target_normal_path> [-b <default_branch>]
git_cmd().args(["clone", "-q", "-b", &default_branch,
    temp_path.to_str()?, target_path.to_str()?]).status()?;
```

### Pattern 6: Default branch detection (Claude's Discretion)

When both `main` and `master` exist, probe in order: (1) `refs/heads/main`, (2) `refs/heads/master`, (3) first branch from `repo.branches(Local)`. Return error if no branches at all.

For bare repos, `Repository::head()` on the bare repo itself returns the symbolic HEAD (typically pointing to `refs/heads/main` or `refs/heads/master`).

```rust
fn detect_default_branch(repo: &Repository) -> Result<String, WorkpotError> {
    // Prefer HEAD symbolic target
    if let Ok(head_ref) = repo.head() {
        if let Some(name) = head_ref.shorthand() {
            return Ok(name.to_string());
        }
    }
    // Fall back: first local branch alphabetically
    let mut branches = repo.branches(Some(git2::BranchType::Local))
        .map_err(|_| WorkpotError::GitUnavailable(...))?;
    if let Some(Ok((b, _))) = branches.next() {
        if let Ok(Some(name)) = b.name() {
            return Ok(name.to_string());
        }
    }
    Err(WorkpotError::InvalidPath("no branches found — cannot determine default branch".into()))
}
```

### Pattern 7: `{worktree}` sanitization with collision hash

[CITED: D-04 from CONTEXT.md — `{worktree}` maps `/` → `.`; collision disambiguated with hash suffix]

```rust
pub fn sanitize_worktree(branch: &str) -> String {
    branch.replace('/', ".")
}

pub fn unique_worktree_name(branch: &str, existing: &[String]) -> String {
    let base = sanitize_worktree(branch);
    if !existing.contains(&base) {
        return base;
    }
    // Collision: append first 6 chars of sha256 of branch name
    use std::hash::{Hash, Hasher};
    let mut h = std::collections::hash_map::DefaultHasher::new();
    branch.hash(&mut h);
    let hash = format!("{:x}", h.finish());
    format!("{}.{}", base, &hash[..6])
}
```

**Note on hash choice:** Claude's Discretion allows any short deterministic suffix. `DefaultHasher` is fine for collision disambiguation (not cryptographic). Length 6 is enough to make collisions essentially impossible in practice.

### Anti-Patterns to Avoid

- **DELETE + re-INSERT for catalog path swap:** Loses untouched columns silently. Use `UPDATE repos SET path=?new WHERE path=?old` instead.
- **Assuming FK ON DELETE CASCADE cascades on UPDATE:** It does not in this schema. Update `repo_tags.repo_path` explicitly in the same transaction.
- **Using git2 for clone operations:** git2 clone API requires a remote URL, not a local path in bare mode. Use `std::process::Command` calling `git clone`, consistent with `common/mod.rs` test helpers.
- **Opening Repository immutably for stash check:** `Repository::stash_foreach` requires `&mut self`. Open a single mutable `Repository` and reuse for all preflight checks.
- **Silently proceeding past rename failure:** If `fs::rename(original, temp)` fails, abort immediately — there is nothing to recover and no state has changed.
- **Updating DB before health check passes:** Health check comes before DB mutation. Order: rename → clone → health → DB → optional delete.
- **Assuming `repo_tags` cascades path update:** Must manually `UPDATE repo_tags SET repo_path = ?new WHERE repo_path = ?old` in same transaction as `UPDATE repos`.

## Common Pitfalls

### Pitfall 1: `repo_tags` FK does not cascade on path UPDATE

**What goes wrong:** After `UPDATE repos SET path = ?new WHERE path = ?old`, tags appear to vanish — they are still keyed to the old path string.
**Why it happens:** `ON DELETE CASCADE` fires on DELETE, not on UPDATE of the referenced PK. This schema does not declare `ON UPDATE CASCADE`.
**How to avoid:** In the same transaction, run `UPDATE repo_tags SET repo_path = ?new WHERE repo_path = ?old` immediately after the repos UPDATE.
**Warning signs:** `list_repos()` returns a record with `tags: vec![]` for a repo that had tags before migration.

### Pitfall 2: `stash_foreach` requires `&mut Repository`

**What goes wrong:** Code compiles fine reading git2 docs superficially, then fails at borrow check because preflight opens `Repository` immutably and then tries to call stash_foreach.
**Why it happens:** `stash_foreach` is declared `&mut self` in git2 because libgit2 internally modifies state during stash iteration.
**How to avoid:** Open `Repository::open_ext` once per preflight call with a mutable binding; chain all checks on the same mutable handle.
**Warning signs:** `error[E0596]: cannot borrow ... as mutable` at compile time.

### Pitfall 3: Clone into an already-existing directory

**What goes wrong:** `git clone ... <target>` fails because the directory was created (e.g., by `create_dir_all`) before the clone.
**Why it happens:** `git clone` refuses to clone into a non-empty directory (and even an empty one with some git tooling).
**How to avoid:** Do NOT `create_dir_all` for clone targets. Let `git clone` create the directory. Only create intermediate parent directories if necessary.
**Warning signs:** Clone subprocess exits with non-zero and stderr contains "already exists and is not an empty directory".

### Pitfall 4: Conversion leaves `.temp` dir orphaned after partial failure

**What goes wrong:** Clone succeeds but health check fails; code deletes `.temp` and the user loses their repo.
**Why it happens:** Confusion in error path — `migration.delete_original` should only fire on full success.
**How to avoid:** D-12 is the rule: on any failure after rename, leave `.temp` in place. Only run filesystem delete (if `delete_original=true`) after DB has been successfully updated.
**Warning signs:** Test: simulate health-check failure and assert `.temp` still exists.

### Pitfall 5: Normal repo worktree path collision (branch name with slashes)

**What goes wrong:** Two branches `feature/login` and `feature/logout` both sanitize to `feature.login` and `feature.logout` — these are distinct, but if you have `feature/lo.gin` and `feature/lo-gin`, after replacing `/` → `.` they become `feature.lo.gin` and `feature.lo-gin` (still distinct). The actual collision case: same sanitized form.
**Why it happens:** `/` → `.` can cause two different branch names to produce the same path segment.
**How to avoid:** `unique_worktree_name` checks existing names and appends hash suffix on collision. Only the initial (current branch) worktree is created during normal→bare conversion (D-10), so this primarily matters in dry-run path display.
**Warning signs:** Dry-run output shows duplicate worktree paths.

### Pitfall 6: Catalog has old path string registered after rename (before clone)

**What goes wrong:** If the process crashes between rename and clone, the catalog still points to the original path (now `.temp`). Next launch shows repo as "missing" because `original` no longer exists.
**Why it happens:** DB is not updated until after successful health check (D-12).
**How to avoid:** This is acceptable per D-12 — `.temp` exists, user can manually recover with `workpot repo add <original>.temp`. Document in SETTINGS.md under "Recovery from interrupted conversion".
**Warning signs:** `missing_repo_paths()` includes the original path after rename.

### Pitfall 7: `branch.name()` returns `Ok(None)` for non-UTF8 branch names

**What goes wrong:** All-branches iterator silently skips non-UTF8 branch names, potentially missing an unpushed branch.
**Why it happens:** `Branch::name()` returns `Ok(None)` for non-UTF8.
**How to avoid:** Treat `Ok(None)` as a branch that cannot be verified — block conversion with a "non-UTF8 branch name detected" preflight error rather than silently skipping.
**Warning signs:** Preflight passes on a repo with a non-UTF8 branch name.

### Catalog path-key swap (in a transaction)

```rust
// Source: design for repo_convert.rs — UPDATE is safer than DELETE+INSERT
pub fn catalog_path_swap(
    conn: &Connection,
    old_key: &str,
    new_key: &str,
    new_name: &str,
    new_git_common_dir: &str,
) -> Result<()> {
    let tx = conn.unchecked_transaction()?;
    tx.execute(
        "UPDATE repos SET path=?1, name=?2, git_common_dir=?3,
         branch=NULL, is_dirty=NULL, ahead=NULL, behind=NULL,
         git_refreshed_at=NULL, git_state_error=NULL
         WHERE path=?4",
        params![new_key, new_name, new_git_common_dir, old_key],
    )?;
    // Must manually cascade because ON UPDATE CASCADE is not declared in the schema
    tx.execute(
        "UPDATE repo_tags SET repo_path=?1 WHERE repo_path=?2",
        params![new_key, old_key],
    )?;
    tx.commit()?;
    Ok(())
}
```

// Source: design for repo_convert.rs
pub fn resolve_template(
    template: &str,
    project: &str,
    worktree: &str,   // empty string when not applicable
) -> String {
    template
        .replace("{project}", project)
        .replace("{worktree}", worktree)
}

pub fn resolve_project_name(config: &MigrationConfig, record: &RepoRecord) -> String {
    match config.project_name_source {
        ProjectNameSource::Alias => record.alias.clone().unwrap_or_else(|| record.name.clone()),
        ProjectNameSource::FolderName => record.name.clone(),
    }
}

The `seed_bare_repo` helper in `tests/common/mod.rs` already covers "bare repo with commits pushed from a seed clone". The missing pieces are: stash creation (use `git_cmd().args(["stash"])`) and bare+worktree setup.

## Open Questions (RESOLVED)

1. **Should `migration.bare_repo_template` and `migration.worktree_template` be validated against path traversal?**
   - What we know: templates are user-controlled strings inserted into path operations.
   - What's unclear: is `../` in a template a security risk in a local-only tool?
   - Recommendation: Validate that resolved target paths are children of the parent directory of the source repo. Reject if resolved path escapes the parent. Add to `MigrationConfig::validate()`.

2. **What happens if the user calls `workpot repo convert` on a worktree of a bare repo (not the bare.git itself)?**
   - What we know: `is_git_worktree` detects linked worktrees; `is_bare_repo` detects bare repos. A linked worktree is a worktree, not a bare repo.
   - What's unclear: should the CLI accept a worktree path and resolve to the bare.git, or reject with "use the bare repo path"?
   - Recommendation: Reject with actionable message: "path is a git worktree; run convert on the bare repo at `<git_common_dir>`". Derive the bare path from `resolve_git_common_dir`.

3. **Hash function for collision suffix (Claude's Discretion)**
   - Recommendation: Use `std::collections::hash_map::DefaultHasher` with 6 hex chars. Deterministic per session (Rust's DefaultHasher is NOT stable across versions, but this is only used for disambiguation display, not for storage keys).
