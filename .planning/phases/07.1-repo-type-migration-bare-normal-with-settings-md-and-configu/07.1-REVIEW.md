# Phase 07.1: Code Review Report

**Reviewed:** 2026-06-10T12:00:00Z
**Depth:** deep
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Deep review traced `workpot repo convert` from CLI (`main.rs`) through `AppContext::convert_repo` into `repo_convert::convert_repo`, preflight helpers in `infra/git.rs`, and `catalog_path_swap`. Core happy-path tests pass (18/18 in `repo_convert_test.rs`), and failure handling correctly avoids DB updates when clone/health-check fails after rename (D-12).

Two correctness gaps threaten post-conversion usability: the catalog records the **bare** path after normal→bare conversion while `launch_repo` / `indexed_launch_path` pass that path directly to `launch_cmd`, so `workpot open` targets a bare repo instead of the linked worktree. Separately, `health_check_bare` uses substring matching on `git worktree list` output, which can false-positive when another worktree path is a prefix-extension of the expected path.

Additional warnings cover missing preflight for target-path collisions (rename happens before clone), branch-name sanitization collisions never checked against on-disk worktrees, and dry-run exiting 0 when preflight is blocked.

## Narrative Findings (AI reviewer)

### Critical Issues

### CR-01: Post-conversion catalog points at bare repo; IDE launch opens wrong path

**File:** `crates/workpot-core/src/services/repo_convert.rs:354-410,451-468`
**Issue:** After normal→bare conversion, `catalog_path_swap` (or the in-place UPDATE branch) stores `bare_git_path` as the catalog `path`. `launch_repo` calls `indexed_launch_path`, which returns that path verbatim and substitutes it into `launch_cmd`. Cursor (or any IDE) is launched on the bare repository directory, not the worktree checkout where developers actually work. The integration test explicitly asserts `repos.iter().any(|r| r.path == to)` where `to` is the bare path, so this is implemented behavior—but it breaks Workpot's core open-in-IDE flow for converted repos.
**Fix:**
```rust
// Option A: catalog the worktree path after normal→bare
let catalog_path = worktree_path.canonicalize()?;
let catalog_name = worktree_path.file_name()...;
catalog_path_swap(conn, &path_key, &catalog_path.display().to_string(), &catalog_name, &gcd)?;

// Option B: keep bare in catalog but resolve launch path
pub fn indexed_launch_path(conn: &Connection, path: &Path) -> Result<PathBuf> {
    let (repo_path, path_key) = resolve_repo_location(conn, path)?;
    if is_bare_repo(&repo_path) {
        let worktrees = git::list_worktree_paths(&repo_path)?;
        return worktrees.into_iter().next()
            .ok_or_else(|| WorkpotError::ConversionPreflight("bare repo has no linked worktree".into()));
    }
    Ok(repo_path)
}
```

### CR-02: `health_check_bare` uses substring match; can pass when expected worktree is missing

**File:** `crates/workpot-core/src/services/repo_convert.rs:244-252`
**Issue:** The health check verifies the worktree exists via `listing.contains(wt.to_str().unwrap_or(""))`. Porcelain output is unstructured text; if another worktree path merely **contains** the expected path as a substring (e.g. expected `/proj/wtrees/main` vs existing `/proj/wtrees/main-backup`), the check passes even when the intended worktree was never created. A failed `git worktree add` could be masked.
**Fix:**
```rust
fn health_check_bare(bare_path: &Path, worktree_path: &Path) -> Result<()> {
    // ... rev-parse HEAD unchanged ...
    let wt = worktree_path.canonicalize().unwrap_or_else(|_| worktree_path.to_path_buf());
    let wt_str = wt.to_str().ok_or_else(|| WorkpotError::ConversionFailed("worktree path not UTF-8".into()))?;
    let mut found = false;
    for line in listing.lines() {
        if let Some(path) = line.strip_prefix("worktree ") {
            if path == wt_str {
                found = true;
                break;
            }
        }
    }
    if !found {
        return Err(WorkpotError::ConversionFailed(
            "health check failed: expected worktree missing".into(),
        ));
    }
    Ok(())
}
```

### Warnings

### WR-01: Target paths not checked for existence before destructive rename

**File:** `crates/workpot-core/src/services/repo_convert.rs:336-348`
**Issue:** Only `temp_path` is checked for prior existence. `bare_git_path`, `worktree_path`, and bare→normal `target_path` are not validated. `std::fs::rename` moves the source to `.temp` first; if `git clone` or `git worktree add` then fails because the destination exists, the user is left with a `.temp` directory and (for partial bare clone success) orphan artifacts, with the catalog unchanged. Recovery is manual per D-12, but the failure is avoidable.
**Fix:** After `resolve_target_paths`, before `rename`:
```rust
for (label, p) in &resolved_paths {
    if label != "temp" && p.exists() {
        return Err(WorkpotError::ConversionPreflight(format!(
            "{label} path already exists: {}; remove or adjust templates",
            p.display()
        )));
    }
}
```

### WR-02: Branch sanitization collisions not detected against existing worktrees

**File:** `crates/workpot-core/src/services/repo_convert.rs:514-516,556-558`
**Issue:** `unique_worktree_name` is always called with `existing: &[]`. Two distinct branches that sanitize identically (e.g. `feature/x` and `feature.x` → both `feature.x`) produce the same worktree directory name. `git worktree add` fails after a successful bare clone, leaving `.temp` and a new bare repo on disk with no catalog update.
**Fix:** Before conversion, list existing worktree directory names under the resolved parent (or query `git worktree list` if converting an already-bare repo) and pass them to `unique_worktree_name`. Add preflight rejection when sanitized names collide and hash disambiguation is required.

### WR-03: Unborn / empty repos pass preflight but fail mid-conversion

**File:** `crates/workpot-core/src/services/repo_convert.rs:79-126,512-513`
**Issue:** `run_preflight` does not reject repos with no commits. `open_and_query` reports branch `"unborn"` (`BRANCH_UNBORN`). Conversion proceeds through rename, then `git worktree add … unborn` fails. Same class of post-rename failure as WR-01.
**Fix:** In `run_preflight` or before bare conversion:
```rust
let state = git::open_and_query(&canonical)?;
if state.branch.as_deref() == Some(crate::domain::BRANCH_UNBORN) {
    return Ok(PreflightResult::NoUpstream { branch: "unborn".into() });
    // or a dedicated PreflightResult::UnbornBranch variant
}
```

### WR-04: `--dry-run` exits 0 when preflight gate fails

**File:** `crates/workpot-cli/src/main.rs:272-295`, `crates/workpot-core/src/services/repo_convert.rs:309-318`
**Issue:** When preflight fails and `dry_run` is true, `convert_repo` returns `Ok(ConvertResult::DryRun { preflight: DirtyWorktree | … })`. CLI prints the result and exits 0. Scripts cannot distinguish "would convert" from "would be blocked" without parsing `preflight:` output.
**Fix:** Either return `Err(ConversionPreflight(...))` on non-Ready preflight even in dry-run, or add a distinct exit code / stderr line and document the contract in SETTINGS.md.

### WR-05: Partial failure after bare clone leaves orphan bare repo on disk

**File:** `crates/workpot-core/src/services/repo_convert.rs:352-401,441-448`
**Issue:** If `git clone --bare` succeeds but `git worktree add` or `health_check_bare` fails, the new bare repository remains on disk alongside `.temp`. D-12 covers catalog/DB safety but not filesystem cleanup guidance for the partial bare artifact.
**Fix:** On worktree-add or health-check failure, attempt `std::fs::remove_dir_all(&bare_git_path)` (when safe) before returning the error, or document explicit cleanup steps in SETTINGS.md recovery section.

### Info

### IN-01: Dead `PreflightResult` variants never produced by `convert_repo`

**File:** `crates/workpot-core/src/services/repo_convert.rs:32-37,296-303`
**Issue:** `PreflightResult::NotInCatalog` and `WrongLayout` are defined and handled in `preflight_message`, but `convert_repo` returns `WorkpotError::ConversionPreflight` for those cases instead. Dead enum arms add maintenance noise.
**Fix:** Remove unused variants or route all preflight outcomes through the enum for consistent CLI formatting.

### IN-02: `find_record` loads entire catalog to match one path key

**File:** `crates/workpot-core/src/services/repo_convert.rs:180-184`
**Issue:** O(n) scan over all repos per conversion. Not incorrect, but scales poorly past hundreds of repos.
**Fix:** Add `catalog::get_repo_by_path(conn, path_key)` using a direct SQL `SELECT`.

### IN-03: SETTINGS.md omits catalog-registration requirement

**File:** `SETTINGS.md:113-121`
**Issue:** Conversion requires the repo to be registered (`find_record` fails otherwise), but SETTINGS.md only documents path, `--to`, and `--dry-run`. Users attempting convert on a discovered-but-unregistered path get a generic preflight error.
**Fix:** Add a bullet under "Conversion command": "Repository must be registered (`workpot repo add`) before conversion."
